# 题库创建流程（审核与发布版）

适用于题目编写/审核/发布的协同流程，面向教研与运营管理。

## 角色划分（建议）

- 题目作者：编写题面、测试点、题解
- 审核人员：复核正确性与可读性
- 发布人员：负责上线与可见性控制

## 全流程

### 1. 题目创建（作者）

入口：`/admin/problems`

步骤：
1. 创建题目基础信息（标题、难度、标签、来源）
2. 可见性先设为 `private` 或 `hidden`
3. 创建首版题面

产出：
- 题目 ID
- v1 题面

### 2. 测试点编制（作者）

入口：`/admin/problems/[id]`

步骤：
1. 添加测试点（输入、输出、分值、分组、样例）
2. 必须覆盖：极限、边界、随机、易错

检查清单：
- 所有样例与题面一致
- 没有“不可判定输出”
- 评分总分为 100 或符合分组策略

### 3. 题解编写（作者）

入口：`/admin/problems/[id]`

步骤：
1. 添加题解（至少一份官方解）
2. 写清：算法思路、复杂度、关键细节

### 4. 初审（审核人员）

建议流程：
1. 自测测试点（用参考答案跑）
2. 人工复核样例
3. 检查是否存在歧义（描述、输入输出）
4. 若发现问题，要求作者新建版本修正

### 5. 复审与定稿（审核人员）

当题目通过初审：
1. 确认题面、测试点、题解一致
2. 标记为“可发布”

### 6. 发布（运营/管理员）

入口：`/admin/problems`

步骤：
1. 将题目可见性改为 `public`
2. 若为竞赛题：设置 `contest`

## 版本策略（推荐）

- 所有修改题面/测试点都必须新建版本
- 不覆盖旧版本
- 旧提交继续绑定历史版本

## 风险控制（强烈建议）

1) 题面与测试点不一致  
→ 强制复核样例 + 参考解跑测  

2) 隐性数据泄漏  
→ 样例与测试点分离，避免敏感数据对外展示  

3) 题目重复/相似  
→ 上线前做“标题/来源/标签”重复扫描  

## 发布后的维护

1. 反馈问题：统一记录问题单  
2. 修复：新建版本，保留旧版本  
3. 重要修复：标注公告
